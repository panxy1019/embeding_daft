Your current approach (source + fallback serialization metadata) is already close to what many dynamic-tool systems evolve toward. The key improvement is to **formalize the storage and recovery layers** so that tool portability is not “best effort” but **deterministic with progressive degradation**.

Below is a practical **Function Capsule Architecture** designed for:

* dynamic or static tools
* portable execution
* progressive fallback
* minimal overhead for simple functions
* automatic escalation for complex tools

The core idea is:

> Store multiple *execution strategies* inside a capsule and let the runtime pick the first valid one.

# 1. Concept: Multi-Layer Function Capsule

A capsule is a **portable execution artifact**.

```
Function Capsule (.fcap)
│
├─ manifest.json
├─ schema.json
│
├─ layer_source/
│   └─ function.py
│
├─ layer_ast/
│   └─ function_ast.json
│
├─ layer_bytecode/
│   └─ function.code
│
├─ layer_file_snapshot/
│   └─ source_bundle.tar
│
├─ layer_wasm/        (optional)
│   └─ module.wasm
│
└─ layer_runner/
    └─ start_server.py
```

The runtime attempts recovery **from top to bottom**.

---

# 2. Recovery Cascade (execution strategy)

Example cascade:

```
1  source reconstruction
2  AST reconstruction
3  bytecode reconstruction
4  file snapshot import
5  wasm runtime
6  embedded runner server
```

Execution algorithm:

```
for layer in capsule.layers:
    try:
        return layer.execute()
    except:
        continue
```

This gives resilience across:

* Python version mismatch
* dependency drift
* moved files
* environment change

Treat a capsule as a **filesystem artifact**.

Example:

```
vector_norm.fcap/
│

├─ schema.json
│
├─ layers/
│   ├─ source/
│   │   └─ function.py

Convert the folder to a **single archive**.

Recommended order of preference:

| format    | reason                     |
| --------- | -------------------------- |
| `tar.zst` | fastest + best compression |
| `tar.gz`  | universal                  |
| `zip`     | easiest on Windows         |

Example:

```
vector_norm.fcap.tar.zst
```

This keeps the capsule **portable and atomic**.

Example pipeline:

     ↓

     ↓

     ↓

Serialized payload example:

Advantages:

# 4. Efficient Transfer Between Machines

When transferring capsules between machines:

Prefer **binary transport**, not base64.

Example:

```
capsule registry
     │

     │
vector_norm.fcap.tar.zst

Base64 should be used only when:

* embedding in JSON
* storing in relational DB
* sending via LLM messages

Otherwise it adds ~33% size overhead.
